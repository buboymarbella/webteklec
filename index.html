HTTP

HTTP Methods

  • GET - Requests data from the resource. Can be cached and remain the browser history.
  • POST - Submits data to the resource. Cannot be cached and process won’t be saved in the browser history.
  • HEAD - Same as GET but only returns HTTP headers
  • PUT - Uploads representation of the specified URI
  • DELETE - Deletes specified resource
  • OPTIONS - Returns HTTP methods that the server supports
  • CONNECT - Converts the request connection to a transparent TCP/IP tunnel

HTTP Message Header Fields:

  • Cache-Control - is used by all caching mechanisms to indicate directives that must be followed along a request and response chain. The directives indicated by the Cache-control header field can override the default caching alorithms used by a server or client.
  • Connection - The connection header field allows the sender to indicate what are desired for the connection to be made. The value of the connection header field is called the connection token.
  • Date - Represents the date and time at which the response or request was created.
  • Pragma - Pragma is used to indicate implementation-specific directives. The directives specify certain behavior from the protocol although some needs to be consistent.
  • Trailer - If the Trailer header field is present it means that the client is willing to accept trailer fields.
  • Transfer-encoding - This header field indicates the type of modification or transformation used on the message body in order to send it to the reciever.
  • Upgrade - Specifies what the client is capable of handling and requests the switch to a much higher version or a different protocol. The server responds with a 101(Switching Protocols) code if possible.
  • Via - Works much like the "Upgrade" header field. It states the additional communication protocols that the client is capable of handling. If the server finds the request as viable it will respond with a 101(Switching Protocols) code.
  • Warning - Used to specify changes in the message which may not be reflected in the message as it is rendered. It states additional information regarding the transformation of the body.
  • Accept - used to specify media types that are acceptable for the response and indicates that the request is fixed to a small set of desired types.
  • Accept-charset - indicates character sets that are acceptable for the response and allows users capable of learning special character sets to signal that capability to a server which can be used for representing documents in a specific character set.
  • Accept-encoding - similar to accept, the difference is that it restricts the codings that are acceptable in the response.
  • Accept-Language - similar to accept, the difference is that it restrics the languages that are acceptable in the response.
  • Authorization - a user agent that authenticates itself usually with a server but it is unnecessary to do so. This field consists of credentials containing the authentication information of the user agent.
  • Expect - indicates particular behaviors of the server that are required by the user.
  • From - this should contain an e-mail address of the user who manipulates the requesting user agent.
  • If-Match - this field is used with a method to make it conditional where a user has one or more entities involved or received from the resource.
  • If-Modified-Since - is used with a method to make it conditional: if the request was not modified since the said time in this field, no entities will be returned from the server i.e. If-Modified-Since: Sat, 29 Oct 1994 19:43:31 GMT
  • If-None-Match is used where a user that has one or more entities received from the resource can verify that none of the them are current by including a list of their entity tags that are associated in the header field.
  • If-Range - is used with the GET method that if a user has a partial copy of an entity in its cache and that wishes to have an updated copy of the entity.
  • If-Unmodified-Since - similarly like if-modified-since, it only sends a response that the entity has not been modified and works only when the server has last modified values associated with the object.
  • Max-forward - is used to limit the number of gateways that can forward the request to the next server.
  • Proxy-Authorization - request header contains the credentials to authenticate a user agent to a proxy, concerned with the status 407 Proxy Authentication that the server responded.
  • Range - indicates a specific part of a document or object the the server should return. Many parts can be requested at the same time and may be returned in a document that has multiple parts.
  • Referrer - a request header that contains the address of the previous web page that contains the link of the current page requested.
  • User-Agent - is a request header which identifies the client, it tells the server what client software is being used, or browser is being used.
  • Accept-Ranges - it is a marker in the Response HTTP header. It advertise its support of partial request that is used by the server. With the presence of this header, it will try to continue the interrupted download rather than to install it from the beginning.
  • E-Tag - entity tag that is part of the HTTP, it is a protocol for the World Wide Web. It is one of several mechanisms that HTTP provides for web cache validation, that allows a client to make conditional requests.
  • Location - it is a header field that is returned in response from an HTTP server in two conditions one is a) to ask a web browser to load a different web page and b) to provide information about the location of a newly created resource.
  • Proxy-Authenticate - it is a request-header field that allows the client to identify or verify itself or its user to a proxy which requires authentication
  • Retry-After - it is a response in HTTP header that indicates how long the user agent should wait before making a request again or follow-up request
  • Server - t is a simple zero – configuration command-line. It is powerful enough for production usage, but it is very simple.
  • Vary - it is a response header that determines how to handle or match future request headers to decide whether a cached response can be used or it needed to request a new one from the origin server.
  • WWW-Authenticate - it is a response header that shows the authentication method to gain access to a resource
  • Allow - This entity header lists all the methods(i.e. GET, PUT, DELETE) that the user-agent is capable of using in a given resource. It is stated to inform the user of what methods is available for use. It is a must to include this header in a 405(Method Not Allowed) response statement.
  • Content-encoding - This entity header field is used to specify to the browser what type of compressing method is used in the resource file. As some files are too large to be sent in their original media-type. This field indicates what type of encoding was applied to identify the decoding to be used in order to get the file extension specified by the media-type field. i.e. Content-Encoding: zip Content-Encoding: gzip
  • Content-language - This entity header indicates the language used in the resource. This may also indicate what language the resource may be displayed in as a Content-Language entity header may have two or more values. The purpose of the header is to inform the user of what audience the entity is intended for. i.e. Content-Language: en, rus, fil;
  • Content-length - This is used to indicate the full size of the entity body in decimal number of octets. Should the user use GET method to get a part of the resource the Content-Length would indicate the full size of the resource. The valid value for this entity header is 0 and above. i.e. Content-Length: 350;
  • Content-location - This is provided by the server to supply the resource location or where the file was fetched from. A server should send the Entity header specially if the resource has multiple associations with it in varying locations because an entity may be accesible through a different URI. i.e. Content-Location: https://www.webtek.org/files/info.txt
  • Content-MD5 - This header field may be generated by a server or client to serve as an integrity check for the resource body. To not defeat the purpose of the Message Integrity Check(MIC) only origin servers and clients are able to generate the tag. An example or one way of checking the integrity of the file is by the number of even or odd bytes in the resource body. If the sent resource byte changed from odd to even or vice-versa the resource is then requested again.
  • Content-range - This entity header field specifies what part of the resource should be fetched by the browser. The user should specify the range to be fetched. To be able to execute the user must follow the correct syntax in specifying the range. i.e. Content-Range = bytes(specification) 500-1000(range)/2000(total);
  • Content-type - This header field states what type of content was sent to the browser. This provides information to the browser on what type of file it is recieving for the browser to be able to render the file properly in its original format. i.e. Content-Type: text/html;
  • Expires - This entity header field indicates the date and time at which the server response is considered stale. A response that is already expired has an Expires entity header field equal to the date header value. For a none expiring response the server sends an Expires entity header field with a value 1 year ahead of the current date/time. i.e. Expires: Mon, 19 Jun 2017 23:59:59 PT;
  • Last Modified - This entity header field indicates the time and date the server thinks the resource was last modified. This is necessary in order for the browser to know if it needs to fetch the resource again due to an update or use the cached version of the resource to save the trouble of fetching the same file again. i.e. Last-Modified: Sun, 18 Jun 2017 08:59:15 PT

HTTP STATUS CODES

  • 1xx: INFORMATIONAL
    • 100: Continue
    • 101: Switching Protocols
  • 2xx: SUCCESS
    • 200: OK
    • 201: Created
    • 202: Accepted
    • 203: Non-Authoritative Information
    • 204: No Content
    • 205: Reset Content
    • 206: Partial Content
  • 3xx: REDIRECTION
    • 300: Multiple Choices - This status code corresponds to any set of representations each with its own specific location
    • 301: Moved Permanently - The resource has been moved to a different location and to a new permanent URI
    • 302: Found - The resource resides temporarily in a different URI.
    • 303: See Other - The resource can be found in a different URI but can only be retrieved using a GET method on said resource. This is like the 302 code only with a restriction.
    • 304: Not Modified - This status code will appear when the user accesses a resource and that it hasn’t been modified.
    • 305: Use Proxy - Only a given proxy will be allowed to access the resource.
    • 306: (UNUSED) -Reserved status code.
    • 307: Temporary Redirect - The resource is temporarily in a different URI since the original location might be altered or deleted.
  • 4xx: CLIENT ERROR
    • 400: Bad Request - The server cannot process the request of the user due to a malformed syntax.
    • 401: Unauthorized - The user doesn’t have any right to access the said location without a valid authorization.
    • 402: Payment Required - The user need to pay a certain amount to gain access to the resource.
    • 403: Forbidden - Access to the resource isn’t possible even with the valid credentials of the user
    • 404: Not Found - The server cannot find any matching URI for the user on said request.
    • 405: Method Not Allowed - The specified method is not allowed by the resource.
    • 406: Not Acceptable - The resource is only capable of generating a response which have content characteristics not acceptable according to the accept headers.
    • 407: Proxy Authentication Required - Similar with the 401 status code but need the user to access a proxy to authenticate itself.
    • 408: Request Time-out - The user exceeded the maximum idle time of the resource.
    • 409: Conflict – There is a conflict with the current state of the resource being asked by the user.
    • 410: Gone - Resources are no longer available in the server.
    • 411: Length Required - Refused to accept a request without a defined content length.
    • 412: Precondition Failed - When the result is false then tested on the server.
    • 413: Request Entity Too Large - The request of the user is too large to handle by the server.
  • 5xx: SERVER ERROR
    • 500: Internal Server Error – The server encounter a condition that prevents for fulfilling the request.
    • 501: Not Implemented – The server does not support/recognise the methods so it lacks to fullfill the request.
    • 502: Bad Gateway – Server received an invalid response from the upstream server that was accessed to attempt to fullfill the request.
    • 503: Service Unavailable – The server may be currently unavailable to handle the request because of the overloading or by having maintenance of the server temporarily.
    • 504: Gateway Timeout – The server did not receive a timely response from the upstream server specified by URI.
    • 505: HTTP Version Not Supported – The Server does not support or it may refuse to support of the HTTP version.
    • 506: Variant Also Negotiates – The variant resource is not a proper end point in the negotiation process.
    • 507: Insufficient Storage – This status code means the server is unable to store the data needed to complete the request.
    • 508: Loop Detected – It indicates that the server encountered an infinite loop while it is processing a request.
    • 509: Bandwidth Limit Exceeded – Server is unavailable temporarily due to the site owner reaching the bandwidth limit.
    • 510: Not Extended – Extension has not been met in the request, so the server send back all the information for the client for that extended request.
    • 511: Network Authentication Required – Client needs to authenticate in order to have network access. Other have used to control over the network like a used to require an agreement to terms of service before granting internet access.
    • 598: Network read timeout error – It is used by some HTTP proxies to signal a network read timeout. This status code is also not specified in any RFC. RFC is Remote Function Call that is a formal document from the internet engineering task force.
    • 599: Network connect timeout error – Also it’s not specified by any RFC. Also used by HTTP proxies to signal network connect timeout.
